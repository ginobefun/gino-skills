# 播客脚本撰写指南

## 整体定位

BestBlogs 每日早报播客是一档**独白式科技资讯播客**，由主播（用户克隆声音）每天用 10-12 分钟的时间，分享当日最值得关注的技术内容。

**核心气质**: 像一位懂技术的朋友在咖啡馆和你聊今天看到了什么有意思的东西——专业但不学术，有观点但不说教，轻松但有信息密度。

---

## 脚本结构

### 总体节奏

```
开场白        ～30秒     轻松破冰，建立今日主题
─────────────────────────
精讲 Top 1    ～2-3分钟   最重要的内容，讲深讲透
精讲 Top 2    ～2-3分钟   第二重要，同样深入
精讲 Top 3    ～2-3分钟   第三重要
─────────────────────────
速览 4-10     ～3-4分钟   7条快速过，每条15-20秒
─────────────────────────
结尾          ～20秒     总结 + 品牌引导
```

**预计总时长**: 10-12 分钟，约 3500-4500 字脚本

---

## 各段落撰写规范

### 1. 开场白（～30秒，100-150字）

**模板**:
```
大家好，我是 {name}，欢迎收听 BestBlogs 每日早报。今天是 {date}。

今天我们聊聊 {关键词1}、{关键词2} 和 {关键词3}。
{一句话预告最吸引人的内容}，我们马上开始。
```

**规则**:
- `{name}` 从 EXTEND.md 的 `podcast_intro_name` 读取
- 关键词来自早报的 3-5 个关键词，选最吸引人的 3 个
- 预告句要具体有悬念，避免 "今天内容很精彩" 这种空话
- **好例子**: "OpenAI 刚刚发布了一个让所有人意外的模型，我们先从这说起。"
- **坏例子**: "今天有很多重要内容，让我们一起来看看。"

### 2. 精讲区（每条 2-3 分钟，500-800 字）

每篇精讲遵循 **BCPT 结构**:

| 段落 | 内容 | 字数 | 说明 |
|------|------|------|------|
| **B**ackground 背景 | 介绍来源、作者、文章背景 | 80-120字 | 建立上下文 |
| **C**ore Problem 核心问题 | 这篇文章要解决/讨论什么 | 50-80字 | 明确读者为什么要关注 |
| **P**oints 核心观点 | 2-3 个最重要的观点或发现 | 250-400字 | 主体内容，信息密度最高 |
| **T**akeaway 思考启发 | 个人评论或对读者的启发 | 80-120字 | 增加个人视角和温度 |

**示例**:

```
第一条，我们来聊聊 Anthropic 刚发布的 Claude Code 新特性。

这篇文章来自 Latent Space 博客，作者是 swyx，他一直在追踪 AI 编程工具的发展。

文章讨论的核心问题是：AI 编程助手如何从"代码补全"进化到"项目级自主开发"。
这其实是整个 AI Coding 领域最热的话题——我们到底能信任 AI 到什么程度。

作者提出了三个关键观点。

第一，Claude Code 的 Agent 模式已经可以独立处理跨文件的重构任务，
这和之前只能做单文件编辑有本质区别。

第二，关键不是 AI 写代码的能力，而是 AI 理解项目上下文的能力。
作者举了一个例子：当你告诉 Claude 改一个 API 的返回格式，
它能自动找到所有相关的测试用例并更新——这需要对整个代码库的理解。

第三，未来的方向可能是"AI 写代码，人类做 Code Review"的工作模式。

我觉得这个观察很有价值。作为一个每天用 Claude Code 的人，
我最大的感受是——它改变的不是编程本身，而是你思考问题的粒度。
你不再需要关注如何实现，而是专注于该实现什么。

详细内容我把链接放在了节目的文字说明里，感兴趣的朋友可以去看原文。
```

**精讲写作要点**:
- 观点来自 `mainPoints` 字段，用自己的话重新表述
- 金句（`keyQuotes`）可以适当引用，用 "作者说……" 引入
- 思考启发部分要有个人视角，不要只是复述原文
- 每条精讲之间用自然过渡，避免机械 "接下来我们看第二条"

**按内容类型调整精讲方式**:

| 类型 | 背景介绍 | 核心素材 | 特殊处理 |
|------|---------|---------|---------|
| **ARTICLE** | 介绍文章来源、作者 | `mainPoints` + Markdown 全文 | 标准 BCPT 结构 |
| **PODCAST** | 介绍播客节目、嘉宾（`speakerSummaries`） | `autoChapters` + `podCastSummary` + `questionsAnswers` | 可引用问答对："主持人问了一个很好的问题——……嘉宾的回答是……" |
| **VIDEO** | 介绍视频创作者、频道 | `mainPoints` + 视频描述 | 配合视频截帧讲解："视频中有个很直观的演示——" |
| **TWITTER** | 介绍推文作者（@用户名）、互动数据 | 推文文本 + 引用的文章/链接 | "这条推文获得了 X 万次浏览和 X 千次转发，{作者}说——" |

- **PODCAST 精讲**: 优先使用 `autoChapters` 组织结构，用 `questionsAnswers` 中的问答作为亮点，`keySentences` 作为金句
- **VIDEO 精讲**: 可适当提及 "视频中演示了……" 等视觉内容，引导听众去看原视频
- **TWITTER 精讲**: 推文通常较短，需要结合推文引用的链接或上下文展开；互动数据（点赞/转推数）可以作为话题热度佐证

### 3. 精讲间过渡

避免千篇一律的过渡词。建议使用的过渡方式:

- **关联式**: "说完了 AI 编程，我们看一个相关的话题——"
- **对比式**: "和上一条不同，这条更偏商业角度——"
- **并列式**: "同样值得关注的是——"
- **时事式**: "这条其实昨天就刷屏了——"

**禁止**: "好的，接下来看第二条"、"下一个"、"然后我们再来看"

### 4. 速览区（7 条，每条 15-20 秒，共 3-4 分钟）

**引导语**:
```
好，以上是今天的三条精讲。接下来快速过 7 条同样值得关注的内容。
```

**每条速览模板**:
```
第{N}条，{标题}，来自{来源}。{一句话核心要点，说明为什么值得关注}。
```

**示例**:
```
第四条，Google 发布 Gemini 2.5 Flash 的 API 更新，来自 Google Developers Blog。
新增了多模态输入和更长的上下文窗口，价格还降了 30%。

第五条，一位独立开发者用 AI 三天搭建了一个完整的 SaaS 产品，来自 Hacker News。
从需求分析到部署上线全程 AI 辅助，他分享了完整的工具链和踩坑经验。
```

**速览写作要点**:
- 使用 `oneSentenceSummary` 字段为基础，适当口语化
- 不需要深入分析，点到为止
- 推文标注 "@用户名"，播客标注 "[播客]"，视频标注 "[视频]"
- 节奏要紧凑，避免冗余词

### 5. 结尾（～20 秒，60-80 字）

**模板**:
```
以上就是今天的 BestBlogs 早报。
今天的关键词是 {关键词1} 和 {关键词2}，
希望这些内容对你有帮助。
我们明天见。BestBlogs，遇见更好的技术阅读。
```

**规则**:
- 简短有力，不要过度客套
- 固定以品牌 slogan "BestBlogs，遇见更好的技术阅读" 结尾
- 可以适当变化前面的话，但品牌结尾保持一致

## 语言风格

### 总体基调

| 维度 | 规范 |
|------|------|
| **正式度** | 半正式——专业但不学术，像技术会议茶歇时的聊天 |
| **人称** | 第一人称 "我"，称呼听众 "大家" 或 "你" |
| **语气** | 好奇、热情、有判断力。不是播报员，而是分享者 |
| **信息密度** | 高——每句话都有信息量，但节奏松弛有度 |

### 口语化处理

TTS 朗读的文本需要口语化处理，避免书面语:

| 书面语 | 口语化 |
|--------|--------|
| 此外 | 另外 |
| 然而 | 不过 |
| 因此 | 所以 |
| 据悉 | 据了解 |
| 上述 | 刚才说的 |
| 值得注意的是 | 有意思的是 |
| 综上所述 | 总的来看 |

### 英文术语处理

- **常见术语保留英文**: AI、API、LLM、Agent、Token、Claude、GPT
- **产品名保留英文**: Claude Code、Cursor、GitHub Copilot
- **技术概念中文优先**: 大语言模型（而非 Large Language Model）、提示词工程（而非 Prompt Engineering）
- **拼写敏感**: TTS 可能误读，必要时加注音提示或避免缩写

### 数字与单位

- 大数字口语化: "一百万" 而非 "1,000,000"，"大约两千" 而非 "2K"
- 百分比: "百分之三十" 而非 "30%"
- 日期: "三月八号" 而非 "3月8日"
- 评分: 不在口播中频繁提及评分数字，影响听感

---

## 禁止事项

- **禁止** 使用 "下面请看"、"让我们来看看" 等主持人套话
- **禁止** 过度使用 "非常"、"特别"、"很" 等程度副词
- **禁止** 每条内容都用 "接下来" 过渡
- **禁止** 结尾说 "感谢收听"（太正式）
- **禁止** 在脚本中使用 Markdown 格式标记（#、**、- 等），纯文本
- **禁止** 在速览区展开讨论——速览就是速览
- **禁止** 背景介绍超过 3 句话——快速进入正题
- **禁止** 脚本总字数超过 5000 字——控制时长

---

## TTS 优化标记

Fish.audio 支持以下文本处理:

- **停顿**: 用 "。" 或 "……" 控制停顿节奏
- **强调**: 适当重复关键词，TTS 会自然加重
- **节奏**: 短句和长句交替，避免单调
- **数字**: 使用中文数字（"三个" 而非 "3个"），归一化会处理但中文更稳定

## 脚本文件格式

脚本保存为 Markdown 文件 `script.md`，使用以下分段标记（供 TTS 脚本分割用）:

```markdown
<!-- SEGMENT: intro -->
大家好，我是 Gino……

<!-- SEGMENT: deep-1 -->
第一条，我们来聊聊……

<!-- SEGMENT: deep-2 -->
说完了 AI 编程，我们看一个相关的话题……

<!-- SEGMENT: deep-3 -->
今天第三条精讲是关于……

<!-- SEGMENT: quick -->
好，以上是今天的三条精讲。接下来快速过七条同样值得关注的内容。
第四条，……
第五条，……
第六条，……
第七条，……

<!-- SEGMENT: quick-2 -->
第八条，……
第九条，……
第十条，……

<!-- SEGMENT: outro -->
以上就是今天的 BestBlogs 早报……
```

分段标记 `<!-- SEGMENT: name -->` 被 `fish-tts.ts` 脚本识别并用于自动分段合成。
