# 播客脚本风格指南

## 核心理念

播客是纯声音媒介。没有画面辅助，100% 的信息靠声音传递。这意味着你的每一句话都必须自己立得住。

**一句话定位**: 一个懂技术的朋友在和你聊他最近看到的有意思的东西。

**关键原则**: 这不是一份填空模板。每期播客的结构应该由内容本身决定。你的播讲应该让听众觉得"这个人真的看懂了这些内容，在帮我提炼价值"。

**风格锚点**: 脚本**必须**读取 `contents/style-profile.md`，注入个人声音特征。以下原则与 style-profile 对齐，style-profile 有更新时以 style-profile 为准。

### 播客的特殊性

| 维度 | 播客 | 视频 |
|------|------|------|
| 信息承载 | 100% 靠声音 | 50-60% 声音 + 40-50% 画面 |
| 描述性语言 | 图表、架构需口述描绘 | 画面展示，配音点到即可 |
| 字数密度 | ~400 字/分钟 | ~300 字/分钟 |
| 精讲字数 | ~500 字/条 | ~350 字/条 |
| 停顿 | 短停顿调节节奏，信息松紧交替 | 可以有静默让画面说话 |

**核心差异**: 视频可以说"你看画面上这个"，播客必须说"我来描述一下这个架构，它大致分三层"。所有信息必须通过声音完整传递。

---

## 三条底线

不管什么类型的播客，这三条必须满足：

1. **内容准确清晰** — 核心信息不失真，技术细节不含糊，该说清楚的一定说清楚
2. **逻辑通顺自然** — 听众不需要倒回去重听就能跟上思路，前后关联清晰
3. **听起来舒服** — 节奏有松有紧，不赶也不拖，像正常人说话

---

## 语言风格

### 声音人格

你不是播报员，不是老师，不是 KOL。你是一个每天都在用 AI 工具做东西的人，分享自己的所见所想。

- 第一人称"我"，称呼听众"大家"或"你"
- 先给结论或判断，再展开为什么这么想
- 多用"我观察到""我最近在试""给我的感觉是"
- 允许说"我不确定""还在验证""先说我的初步判断"
- 遇到好东西，真诚推荐。遇到有争议的，给出自己的看法但留空间
- 适度暴露约束，比如"时间有限""我只看了前半部分"

### 口语化

TTS 朗读需要口语化，书面语直接读会很僵：

| 别用 | 改成 |
|------|------|
| 此外/另外 | 还有一个 |
| 然而/但是 | 不过 |
| 因此/所以 | 所以 |
| 值得注意的是 | 有意思的是 |
| 综上所述 | 总的来看 |
| 在这个方面 | 这块 |
| 进行了探讨 | 聊了聊 |

### 英文术语

- 常见术语保留英文: AI、API、LLM、Agent、Token、Claude、GPT
- 产品名保留英文: Claude Code、Cursor、GitHub Copilot
- 技术概念中文优先: 大语言模型而非 Large Language Model

### 数字

- 大数字口语化: "一百万" 而非 "1,000,000"
- 百分比: "百分之三十" 而非 "30%"
- 日期: "三月八号" 而非 "3 月 8 日"

### 标点（TTS 适配）

- 不用引号，改用语气词或直接说
- 不用破折号，改用句号断句
- 不用括号，补充信息直接用短句展开

## 源材料驱动创作

播客脚本的质量取决于你对源材料的理解深度。脚本生成前，`contents/tmp/podcast/YYYY-MM-DD/materials.md` 应该已经准备好。

### 用好源文章的真实素材

播客没有画面，源材料的利用方式和视频不同。关键是用声音把好内容讲活。

| 素材类型 | 播客脚本中怎么用 |
|---------|--------------|
| 关键数据 | "这篇文章提到一个数字，百分之五十二的开发者...这个数字说明什么呢" |
| 原文金句 | 用语气变化标记引用："原文里有一句话说得特别好......然后引述......我觉得这句话抓住了关键" |
| 架构图/流程图 | 口述关键结构："简单说一下这个系统的架构，它大致分三层。最上面是接口层，中间是...底层是..." |
| 实践案例 | "他们实际跑下来的结果是...具体来说，性能提升了百分之四十" |
| 代码示例 | 不读代码，描述逻辑和思路："这段代码的关键思路是，先检查缓存，如果没命中再去调 API" |

**核心原则**: 源文章中已有好素材，比如数据、案例、金句，直接用。不要扔掉好内容然后用模糊的概括替代。

### 融入历史关联

查阅了近期内容后，适当在脚本中建立关联：

- **延续性**: "上次聊到 Claude Code 的多文件编辑，这次他们又更新了......"
- **验证/反转**: "之前我说过对 XX 持观望态度，现在有了新的数据......"
- **主题积累**: "最近连续好几天都有 Agent 相关的内容，说明这个方向确实在加速"
- **避免重复**: 如果某话题上期已深度覆盖，本期只说增量："关于 XX 我们上期详细聊过了，今天的新信息是......"

**注意**: 关联要自然，不要生硬。一期播客中历史引用 1-2 次足够，多了会让新听众困惑。

### 跨来源综合

当多条内容涉及同一话题时，脚本不应该逐条复述，而应该综合：

```
不好的方式：
"第一篇文章说了 XX。第二篇文章也说了 XX。第三条推文也提到了 XX。"

好的方式：
"最近 XX 这个话题很热。A 从技术角度分析了原因，B 在实际项目里验证了效果，
而 C 指出了一个大家可能忽略的风险。综合来看......"
```

## 叙事策略库

阅读完所有内容后，根据内容特征选择最合适的叙事策略。**同一种输入类型可以用不同策略，甚至混合策略**。

### 策略一：标准精讲 + 速览

**适用**: 多数 daily 场景，内容质量均匀，2-3 条值得展开 | **10-12 分钟 / 3800-5000 字**

```
开场 (30-40秒, ~200字) → 打招呼 + 今天最值得聊的一句话钩子
精讲 2-3 条 (每条 2-4分钟, ~400-500字/条) → 为什么值得聊 → 核心信息 → 有意思的点 → 我的看法
速览 5-7 条 (2-3分钟, ~50-80字/条) → 是什么 + 为什么值得知道
收尾 (20-30秒, ~100字) → 最值得记住的一个点 + 推荐原文
```

**精讲选择**: 不一定选 Top 3。一条特别好可以花 4 分钟，其他各 2 分钟。有关联的内容可以串讲。整体中等时，多选几条各 2 分钟，不硬凑深度。

**选择信号**: 内容间关联性弱，各自独立成立，没有明显主线

### 策略二：深度聚焦

**适用**: 有一条内容特别好，值得花大量时间讲透 | **5-10 分钟 / 2000-4000 字**

```
开场 (20-30秒, ~130字) → 直接抛出话题的核心张力，不绕弯
深度展开 (4-6分钟, ~1600-2500字) → 结论 → 为什么 → 有意思的细节 → 延伸思考。每 2 分钟一个钩子
速览 (1-2分钟, 可选) → 其他内容快速带过
收尾 (15-20秒, ~80字) → 核心判断 + 推荐原文
```

**播讲要点**: 深度不等于长，重点是思考深度。播客无画面，每个论点必须用声音讲清楚，适当多给一句解释。

**选择信号**: 某条评分远超其他，话题复杂度高，或用户指定"重点聊某条"

### 策略三：主题串联

**适用**: 多条内容指向同一话题，串起来比单独讲更有价值 | **10-15 分钟 / 4000-6000 字**

```
开场 (30-40秒, ~180字) → "今天有好几条都和 XX 有关，可以放一起聊"，亮出主题线
主题展开 (8-12分钟) → 围绕主题编织：A 提出问题 → B 给出方案 → C 验证效果。要有递进
补充速览 (1-2分钟, 可选) → 不相关的内容快速带过
收尾 (20-30秒, ~120字) → 主题整体判断 + 下一步值得关注什么
```

**播讲要点**: 串联不是"A 说了 X，B 也说了 X"，而是有递进关系。没有画面提示话题归属，逻辑连接词更重要。

**选择信号**: 2-3 条内容话题重叠，串联后信息增量大于分开讲

### 策略四：对比分析

**适用**: 两条内容形成对比、互补或对立关系 | **10-12 分钟 / 3800-5000 字**

```
开场 (25-35秒, ~150字) → "今天有两个很不同的声音"
A 观点 (2-3分钟, ~800-1200字) → 公平呈现核心观点和依据
B 观点 (2-3分钟, ~800-1200字) → 公平呈现核心观点和依据
对比总结 (1-2分钟, ~500-700字) → 你的判断，可以说各自的适用边界
速览 + 收尾 (2-3分钟)
```

**播讲要点**: 公平性很重要，不要一开始暗示谁对谁错。播客的对话感特别适合"一方面......另一方面......"的结构。

**选择信号**: 两条内容观点相反、方案不同、或同一话题的不同角度

### 策略五：趋势信号

**适用**: 多条内容指向一个正在形成的趋势 | **10-12 分钟 / 3800-5000 字**

```
开场 (25-35秒, ~150字) → 直接亮出趋势判断
信号列举 (2-3分钟, ~120-180字/条) → 3-5 个信号快速摆证据，节奏明快
深入分析 (3-5分钟, ~500-700字/条) → 挑 1-2 个最关键信号展开因果和影响
判断 (1-2分钟, ~400-600字) → 趋势意味着什么？给出可行动建议
收尾 (15-25秒)
```

**播讲要点**: 趋势判断要有边界感，不说"颠覆一切"，而是"对 XX 类型的团队影响最大"。播客无画面列表，用语气节奏制造摆证据的感觉。

**选择信号**: 3+ 条内容说同一件事但角度不同，或有明显行业集体动向

### 策略六：实操拆解

**适用**: 内容偏工具、教程、实战经验 | **5-10 分钟 / 2000-4000 字**

```
痛点引入 (20-30秒, ~120字) → "如果你也遇到过 XX 问题"
背景 (30-60秒, ~250字) → 解决什么问题，之前怎么做
步骤/要点 (3-6分钟, ~1500-2500字) → 做什么 → 为什么 → 踩坑点
效果 (30-60秒, ~250字) → 实际效果 + 局限性
收尾 (15-20秒, ~80字) → 行动建议
```

**播讲要点**: 播客无法展示截图和代码，必须把步骤说清楚。用类比帮助理解。复杂步骤引导听众看原文。

**选择信号**: 内容以 How-to、工具介绍、配置教程、经验分享为主

### 策略选择决策流程

```
阅读所有内容
  ↓
有 3+ 条指向同一趋势？ → 策略五（趋势信号）
  ↓ 否
有 2 条形成对比/对立？ → 策略四（对比分析）
  ↓ 否
有 2-3 条话题重叠可串联？ → 策略三（主题串联）
  ↓ 否
有 1 条远超其他？ → 策略二（深度聚焦）
  ↓ 否
内容偏教程/实操？ → 策略六（实操拆解）
  ↓ 否
→ 策略一（标准精讲 + 速览）
```

**可以混合**: 比如用策略三讲主体 + 策略一的速览收尾。策略只是思路，不是牢笼。

### Weekly 特别说明

周刊播客天然适合策略三（主题串联）或策略五（趋势信号），因为一周的内容通常能归纳出 2-3 个主题线。但如果这周内容很分散，也可以用策略一。

**关键区别**: 周刊不是日报的堆叠，要找到**这一周的叙事线**。

## 开场变化库

**核心要求**: 每期播客的开场不能一样。以下是可选的开场方式，根据内容灵活选用。

| 开场方式 | 示例 | 适用场景 |
|---------|------|---------|
| **钩子先行** | "今天有个东西，我觉得可能会改变 XX 的方式" | 有一条特别值得聊的内容 |
| **反直觉** | "大家都在说 AI Agent，但今天这篇文章给了一个完全不同的视角" | 对比分析或有争议的内容 |
| **个人经历** | "最近我在用 Claude Code 做项目时碰到一个问题" | 实操拆解 |
| **趋势观察** | "这两天刷到好几篇文章都在说同一件事" | 趋势信号 |
| **数据切入** | "一个数字。百分之五十二。这是......" | 有惊人数据的内容 |
| **问题驱动** | "如果 AI 真的能替代百分之五十的工作，最先消失的会是什么？" | 深度聚焦 |
| **轻松日常** | "嘿大家好，今天是周五，信息量不大，但有几条挺有意思" | 内容较轻的日子 |
| **延续上期** | "上次聊到 XX，这两天又有新进展" | 与前期内容有关联 |
| **录播日记** | "今天录这期的时候刚好试完一个新工具，顺便聊聊" | 有个人体验可分享 |

**绝对不要**每天都用: "嘿大家好，我是 XX，今天是 X 月 X 号"。这种开场偶尔用可以，天天用就是噪音。

## 过渡策略库

条目之间的过渡也应该多样化。播客没有画面切换做视觉分隔，过渡必须显式说出，比视频更关键。

| 过渡方式 | 示例 | 适用场景 |
|---------|------|---------|
| **关联** | "说到 AI 编程，还有一个相关的东西" | 相邻内容有话题关联 |
| **转换** | "换个话题。最近......" | 话题完全不同 |
| **对比** | "和上面这个不同，下一篇更偏......" | 相邻内容有对比 |
| **递进** | "其实刚才那个问题，这篇文章给了一个更具体的答案" | 后一条是前一条的深入 |
| **直接** | 不加过渡，直接开始下一条 | 前一条已自然收束 |
| **元评论** | "好，前面两条都偏宏观，下面来一条更接地气的" | 调节节奏 |

**绝对不要**: "好的，接下来看第二条""下面请看""让我们来看看"

## 节奏与信息密度

### 信息密度曲线

播客要有节奏感。不能一直高密度（累），也不能一直低密度（无聊）。

```
密度
 高  ╱╲    ╱╲        ╱╲
    ╱  ╲  ╱  ╲      ╱  ╲
 低╱    ╲╱    ╲____╱    ╲___
   开场  精讲1  精讲2  速览  收尾
```

- **开场**: 中等密度，建立兴趣
- **精讲**: 高密度（核心观点）→ 低密度（个人反思、反问）交替
- **精讲间过渡**: 短暂低密度呼吸
- **速览**: 均匀中低密度，快节奏
- **收尾**: 低密度，留下回味

### 信息松紧交替

播客没有画面分担注意力，纯靠声音的节奏控制。核心机制是**松紧交替**：

- **紧**（高密度）: 核心论点、关键数据、技术细节。句子短，信息密
- **松**（低密度）: 个人感受、反问、类比、短暂停顿。让听众消化

一段连续高密度不要超过 90 秒，之后要给一个"呼吸点"。可以是一句个人感受，也可以是一个反问。

### TTS 停顿控制

脚本中用省略号 `......` 标记需要短停顿的位置：

- 话题切换时停顿：让听众心理分段
- 重要结论前停顿：制造期待
- 金句引述后停顿：让听众消化
- 数据展示后停顿：给理解留空间

## 分享心态

播客的核心目的是**让听众从中获得信息和思考**，不是展示你知道多少。

### 信息传递检查

每条精讲必须回答：
1. **这是什么** — 一句话说清楚核心
2. **为什么重要** — 对谁有什么影响
3. **我的判断** — 你怎么看，有什么可行动的

速览必须回答：
1. **这是什么** — 一句话概括
2. **为什么值得知道** — 为什么进了今天的推荐

### 价值密度原则

- 每 30 秒至少输出一个有价值的信息点
- 避免重复说同一个观点的不同表述
- 如果某段没有新信息，砍掉
- 宁可少说一条，也不要每条都蜻蜓点水

---

## 内容讲述原则

不要套结构，先问自己：**如果我在和一个朋友聊这个，我会怎么说？** 通常是：建立兴趣 → 核心信息 → 有意思的点 → 个人看法。但这不是刚性结构，**让内容决定节奏**。

| 内容特点 | 讲述策略 |
|---------|---------|
| 技术突破/新产品 | 先说改变了什么，再说怎么做到的，最后说对谁有影响 |
| 实践经验/教程 | 先说痛点，再说方案，重点在可复用的部分 |
| 行业观点/趋势 | 先说结论，再说支撑证据，最后说你同不同意 |
| 数据/研究报告 | 挑最反直觉的发现说，不要面面俱到 |
| 推文/短内容 | 说清楚为什么值得展开聊，背后的信号是什么 |
| 播客/视频 | 提炼核心问答或最有洞察的片段，不是全文复述 |

## 分段标记

用 `<!-- SEGMENT: name -->` 标记 TTS 分段点，放在**内容的自然分界处**。标记名称用描述性命名（`deep-claude-code` 比 `deep-1` 好）。

分段模式因策略不同：

| 策略 | 分段模式 |
|------|---------|
| **标准精讲** | `intro → deep-{topic-1} → deep-{topic-2} → quick-browse → outro` |
| **深度聚焦** | `intro → deep-setup → deep-core → deep-reflection → outro` |
| **主题串联** | `intro → theme-{name-1} → theme-{name-2} → quick-browse → outro` |
| **对比分析** | `intro → perspective-a → perspective-b → synthesis → outro` |
| **趋势信号** | `intro → signals → deep-signal-{name} → judgment → outro` |
| **实操拆解** | `intro → background → steps → results → outro` |

## 禁止事项

### 语气

- 恐惧渲染: "不做就会被淘汰""时代抛弃你"
- 宏大空泛: "颠覆一切""重塑世界""革命性变革"
- 说教: "你必须""所有人都该"

## 脚本自检

写完脚本后检查：

- [ ] **内容准确性**: 数据、结论与源文章一致？没有把推测说成事实？
- [ ] **素材利用**: 源文章中的好数据、好案例、好金句有没有用上？
- [ ] **历史关联**: 与近期内容的关联是否自然？不强求，有就加
- [ ] 把脚本从头默读一遍，**像在听**，不像在看。有没有哪里觉得别扭？
- [ ] 这期播客的开场和上一期不一样吗？切入角度有新意吗？
- [ ] 核心信息是否准确？有没有过度简化导致失真？
- [ ] 听完能记住几个关键点？如果一个都记不住，信息密度需要调整
- [ ] 有没有哪里觉得无聊或冗余？删掉
- [ ] 有没有哪里太快听不懂？放慢，多加一句解释
- [ ] 开头 30 秒是否有吸引力？如果没有，听众会关掉
- [ ] 是像一个人在说话，还是像一篇文章在被朗读？
- [ ] 每条精讲都传递了"这是什么、为什么重要、我怎么看"？
- [ ] 叙事策略选择合理吗？有没有更好的方式组织这些内容？
- [ ] 没有恐惧渲染、宏大空泛、说教语气？
- [ ] 没有引号、破折号、括号？数字已转中文口语？
