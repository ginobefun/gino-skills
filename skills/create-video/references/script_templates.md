# 视频配音脚本风格指南

## 核心理念

视频配音不是播客的缩略版。画面承担了 40-50% 的信息传递，配音的职责是**引导注意力、补充语境、输出判断**。

**一句话定位**: 一个每天在做 AI 产品的人，边看画面边和你聊他的理解。

**关键原则**: 这不是一份填空模板。每期视频的结构应该由内容本身决定。你的配音应该让观众觉得"这个人真的看懂了这些内容，在帮我提炼价值"。

**风格锚点**: 脚本**必须**读取 `contents/style-profile.md`，注入个人声音特征。以下原则与 style-profile 对齐，style-profile 有更新时以 style-profile 为准。

---

## 视频 vs 播客：配音差异

| 维度 | 播客配音 | 视频配音 |
|------|---------|---------|
| 信息承载 | 100% 靠声音 | 50-60% 声音 + 40-50% 画面 |
| 描述性语言 | 需要口述图表、数据 | 画面展示，配音点到即可 |
| 字数密度 | ~400 字/分钟 | ~300 字/分钟（留给视觉消化） |
| 停顿 | 短停顿调节节奏 | 可以有 1-2 秒静默让画面说话 |
| 总字数 | 精讲 ~500 字/条 | 精讲 ~350 字/条 |

**核心差异**: 播客需要"说画面"，视频需要"说画面没有的"。

---

## 三条底线

不可妥协：

1. **内容准确清晰** — 核心信息不失真，技术细节不含糊
2. **逻辑通顺自然** — 观众不需要倒回去重看就能跟上
3. **观感舒适** — 节奏有松有紧，声画同步，没有等待感

## 语言风格

### 声音人格

你不是播报员，不是老师，不是 KOL。你是一个每天都在用 AI 工具做东西的人，分享自己的所见所想。

- 第一人称"我"，称呼观众"大家"或"你"
- 先给结论或判断，再展开
- 允许不确定："还在验证""先说初步判断"
- 视频特有：可以用"你看这个""画面上这个"等指示性语言

### 口语化要求

书面语替换为口语。额外注意：

- 画面有文字时不要逐字念，用自己的话概括
- 看到数据图表说"你可以看到"而非复述每个数字
- 遇到代码截图说"这段代码的关键在于"而非念代码

### 情绪标记（Fish.audio S2 Pro）

脚本中在关键位置的**句首**添加情绪标记，增强语音表现力：

```
(thoughtful) 我觉得这个思路特别值得关注。
(excited) 数据上特别有说服力。
(serious) 一个数字。十七点二倍。
```

**使用策略**:
- 每个 SEGMENT 使用 1-2 个情绪标记，不要过度
- 在关键转折点、数据亮点、个人判断处使用
- 常用标记：`(excited)` 亮眼发现、`(thoughtful)` 深度分析、`(serious)` 重要数据
- 标记**必须在句首**，不能放在句中
- 不要混用冲突情绪，不要自创标记

### 标点与排版

**标点清理（脚本 + slides 文字均适用）**:
- 不用引号，改用语气词或直接说
- 不用破折号，改用句号断句
- 不用括号，补充信息直接用短句展开
- 遇到原文中有引号/破折号/括号的内容，改写为自然的口语表达

**中文排版**:
- 中文与英文之间加一个空格：`Claude Code 的新功能` 而非 `Claude Code的新功能`
- 中文与数字之间加一个空格：`超过 500 万用户` 而非 `超过500万用户`
- 数字与单位之间加一个空格：`30 秒` 而非 `30秒`

**英文术语**: 常见术语保留英文（AI、API、LLM、Agent、Token），产品名保留英文（Claude Code、Cursor），技术概念中文优先（大语言模型而非 Large Language Model）

**数字**: 脚本中用中文口语（一百万、百分之三十），slides 文字中可用阿拉伯数字（500 万、30%）

### 表达优化

**简化复杂句式**: 原文中的长难句要拆成短句。观众没有回看的习惯，一句话说不清就拆成两句。

```
❌ 复杂：
"基于混合数据飞轮的方法论，通过大模型合成的复杂多窗口场景配合真实设备自动回测
实现了跨端泛化的稳定性提升。"

✅ 简化：
"他们的方法挺巧妙的。先用大模型合成各种复杂场景，然后在真实设备上自动跑测试。
这样训练出来的模型，换个平台也能用。"
```

**保持信息准确**: 简化表达不等于丢失信息。核心概念、数字、因果关系必须保留。简化的是句式，不是内容。

**适当的情绪和风趣**:
- 遇到真正有意思的内容，可以表达出来："这个思路我觉得挺妙的""说实话看到这个数字我还挺意外的"
- 遇到值得吐槽的，温和地说："这个说法……我个人持保留态度""听起来很美好，但实际操作起来可能没那么简单"
- 不要刻意搞笑，自然流露即可
- 风趣不等于轻浮。在数据和事实部分保持严肃准确

## 源材料驱动创作

配音脚本的质量取决于你对源材料的理解深度。脚本生成前，`contents/tmp/video/YYYY-MM-DD/materials.md` 应该已经准备好。

### 用好源文章的真实素材

| 素材类型 | 脚本中怎么用 | 对应 slide 处理 |
|---------|------------|----------------|
| 关键数据 | "这篇文章提到一个数字，百分之五十二的……" | `point` slide 展示数据 |
| 原文金句 | 停顿让观众读金句，再加你的评论 | `quote` slide 展示 |
| 架构图/流程图 | "你看画面上这个架构图，关键在于……" | `point` slide 用原文图 |
| 实践案例 | "他们实际跑下来的结果是……" | `point` slide 配截图 |
| 代码示例 | "这段代码的关键在于这一行" | `point` slide 配代码截图 |

**核心原则**: 源文章中已有好素材时（数据、图表、案例），直接用。不要把好内容扔掉然后用 AI 生成一张模糊的概念图。

### 融入历史关联

查阅了近期内容后，适当在脚本中建立关联：

- **延续性**: "上次聊到 Claude Code 的多文件编辑，这次他们又更新了……"
- **验证/反转**: "之前我说过对 XX 持观望态度，现在有了新的数据……"
- **主题积累**: "最近连续好几天都有 Agent 相关的内容，说明这个方向确实在加速"
- **避免重复**: 如果某话题上期已深度覆盖，本期只说增量："关于 XX 我们上期详细聊过了，今天的新信息是……"

**注意**: 关联要自然，不要生硬。一期视频中历史引用 1-2 次足够，多了会让新观众困惑。

### 跨来源综合

当多条内容涉及同一话题时，脚本不应该逐条复述，而应该综合：

```
❌ 不好的方式：
"第一篇文章说了 XX。第二篇文章也说了 XX。第三条推文也提到了 XX。"

✅ 好的方式：
"最近 XX 这个话题很热。A 从技术角度分析了原因，B 在实际项目里验证了效果，
而 C 指出了一个大家可能忽略的风险。综合来看……"
```

## 各叙事策略的配音指南

### 策略一：标准精讲 + 速览

**目标时长**: 8-12 分钟 | **目标字数**: 2800-4200 字

```
开场 (25-35秒, ~150字)
  → 打招呼 + 今天最值得聊的一句话钩子

精讲 2-3 条 (每条 2-4分钟, ~250-350字/条)
  → 每条：为什么值得看 → 核心发现 → 我的看法
  → 条目间自然过渡，不机械

速览 5-7 条 (2-3分钟, ~30-50字/条)
  → 每条一句话：是什么 + 为什么值得知道

收尾 (15-25秒, ~80字)
  → 今天最值得记住的一个点 + 推荐原文
```

**精讲选择策略**:
- 不一定选 Top 3。一条特别好可以花 4 分钟，其他各 2 分钟
- 有关联的内容可以串讲，形成小主题
- 整体中等时，多选几条各 2 分钟，不硬凑深度

### 策略二：深度聚焦

**目标时长**: 5-10 分钟 | **目标字数**: 1800-3500 字

```
开场 (20-30秒, ~120字)
  → 直接抛出话题的核心张力，不绕弯

深度展开 (4-6分钟, ~1500-2200字)
  → 层层递进，不是平铺
  → 先说结论 → 为什么这么想 → 最有意思的细节 → 延伸思考
  → 每 2 分钟要有一个"钩子"防止观众划走

速览 (1-2分钟, 可选)
  → 如果有其他内容，快速带过

收尾 (15-20秒, ~60字)
  → 核心判断 + 推荐原文
```

**配音要点**: 深度不等于长。重点是思考深度，不是信息堆叠。如果某段展开后发现没有新信息，砍掉。

### 策略三：主题串联

**目标时长**: 8-15 分钟 | **目标字数**: 2800-5500 字

```
开场 (30-40秒, ~150字)
  → "今天（这周）有好几条都和 XX 有关，我觉得可以放一起聊"
  → 亮出主题线

主题展开 (6-12分钟)
  → 不是逐条讲完再总结
  → 而是围绕主题编织：A 说了什么 → B 补充了什么 → C 提供了另一个角度
  → 关键是：串联后的信息增量大于分开讲

收尾 (20-30秒, ~100字)
  → 这个主题的整体判断 + 下一步值得关注什么
```

**配音要点**: 串联不是"A 说了 X，B 也说了 X"。而是"A 提出了问题，B 给出了方案，C 在实践中验证了"。要有递进关系。

### 策略四：对比分析

**目标时长**: 8-12 分钟 | **目标字数**: 2800-4200 字

```
开场 (25-35秒, ~130字)
  → "今天有两个很不同的声音" 或 "这两篇文章对同一件事的判断完全不同"

A 观点 (2-3分钟, ~700-1000字)
  → 公平呈现 A 的核心观点和依据

B 观点 (2-3分钟, ~700-1000字)
  → 公平呈现 B 的核心观点和依据

对比总结 (1-2分钟, ~400-600字)
  → 你的判断：不一定选边，可以说各自的适用边界
  → "我个人更倾向……因为……但如果你的情况是……那 B 的观点更适用"

速览 + 收尾 (2-3分钟)
```

**配音要点**: 公平性很重要。不要一开始就暗示谁对谁错。让观众在听完两方后形成自己的判断，最后再给你的思考。

### 策略五：趋势信号

**目标时长**: 8-12 分钟 | **目标字数**: 2800-4200 字

```
开场 (25-35秒, ~130字)
  → 直接亮出趋势判断："最近一个明显的信号是……"

信号列举 (2-3分钟, ~100-150字/条)
  → 3-5 个信号，每个快速说：谁说了什么 + 为什么是证据
  → 节奏快，像在"摆证据"

深入分析 (3-5分钟, ~400-600字/条)
  → 挑 1-2 个最关键的信号展开
  → 说清楚因果关系和影响面

判断 (1-2分钟, ~300-500字)
  → 这个趋势意味着什么？对谁有影响？
  → 给出可行动的建议

收尾 (15-25秒)
```

**配音要点**: 趋势判断要有边界感。不要说"颠覆一切"，而是说"对 XX 类型的团队/产品影响最大，因为……"。

### 策略六：实操拆解

**目标时长**: 5-10 分钟 | **目标字数**: 1800-3500 字

```
痛点引入 (20-30秒, ~100字)
  → "如果你也遇到过 XX 问题" 或 "最近我在搭建 XX 时……"

背景 (30-60秒, ~200字)
  → 解决什么问题，之前是怎么做的

步骤/要点 (3-6分钟, ~1200-2200字)
  → 按逻辑顺序展开，配合截图
  → 每步说清楚：做什么 → 为什么这么做 → 踩坑点

效果 (30-60秒, ~200字)
  → 实际效果 + 局限性

收尾 (15-20秒, ~60字)
  → 行动建议
```

**配音要点**: 画面配合是关键。说到哪步，画面就显示那步的截图。"你看画面上这个配置"比"在配置文件里加上以下参数"更适合视频。

## 开场变化库

**核心要求**: 每期视频的开场不能一样。以下是可选的开场方式，根据内容灵活选用。

| 开场方式 | 示例 | 适用场景 |
|---------|------|---------|
| **钩子先行** | "今天有个东西，我觉得可能会改变 XX 的方式" | 有一条特别值得聊的内容 |
| **反直觉** | "大家都在说 AI Agent，但今天这篇文章给了一个完全不同的视角" | 对比分析或有争议的内容 |
| **个人经历** | "最近我在用 Claude Code 做项目时碰到一个问题" | 实操拆解 |
| **趋势观察** | "这两天刷到好几篇文章都在说同一件事" | 趋势信号 |
| **数据切入** | "一个数字。百分之五十二。这是……" | 有惊人数据的内容 |
| **问题驱动** | "如果 AI 真的能替代百分之五十的工作，最先消失的会是什么？" | 深度聊焦 |
| **轻松日常** | "嘿大家好，今天是周五，信息量不大，但有几条挺有意思" | 内容较轻的日子 |
| **延续上期** | "上次聊到 XX，这两天又有新进展" | 与前期内容有关联 |

**绝对不要**每天都用: "嘿大家好，我是 XX，今天是 X 月 X 号"。这种开场偶尔用可以，天天用就是噪音。

## 过渡策略库

条目之间的过渡也应该多样化：

| 过渡方式 | 示例 | 适用场景 |
|---------|------|---------|
| **关联** | "说到 AI 编程，还有一个相关的东西" | 相邻内容有话题关联 |
| **转换** | "换个话题。最近……" | 话题完全不同 |
| **对比** | "和上面这个不同，下一篇更偏……" | 相邻内容有对比 |
| **递进** | "其实刚才那个问题，这篇文章给了一个更具体的答案" | 后一条是前一条的深入 |
| **直接** | （无过渡，直接开始） | 前一条已自然收束 |
| **元评论** | "好，前面两条都偏宏观，下面来一条更接地气的" | 调节节奏 |

**绝对不要**: "好的，接下来看第二条""下面请看""让我们来看看"

## 节奏与停顿

### 信息密度曲线

视频要有节奏感。不能一直高密度（累），也不能一直低密度（无聊）。

```
密度
 高  ╱╲    ╱╲        ╱╲
    ╱  ╲  ╱  ╲      ╱  ╲
 低╱    ╲╱    ╲____╱    ╲___
   开场  精讲1  精讲2  速览  收尾
```

- **开场**: 中等密度，建立兴趣
- **精讲**: 高密度（核心观点）→ 低密度（消化时间）交替
- **精讲间过渡**: 短暂低密度呼吸
- **速览**: 均匀中低密度，快节奏
- **收尾**: 低密度，留下回味

### 声画同步停顿

配音脚本中用省略号 `……` 标记需要短停顿的位置：

- slide 切换时停顿 0.5 秒：让观众适应新画面
- 金句显示时停顿 1-2 秒：让观众读完再评论
- 话题切换时停顿 1 秒：心理分段
- 重要结论前停顿 0.5 秒：制造期待

## 分段标记

用 `<!-- SEGMENT: name -->` 标记 TTS 分段点。

分段策略因叙事策略不同：

**标准精讲**: 每条精讲独立分段
```markdown
<!-- SEGMENT: intro -->
<!-- SEGMENT: deep-topic-name-1 -->
<!-- SEGMENT: deep-topic-name-2 -->
<!-- SEGMENT: quick-browse -->
<!-- SEGMENT: outro -->
```

**主题串联**: 按主题分段（不按条目）
```markdown
<!-- SEGMENT: intro -->
<!-- SEGMENT: theme-agent-infra -->
<!-- SEGMENT: theme-ai-coding -->
<!-- SEGMENT: quick-browse -->
<!-- SEGMENT: outro -->
```

**对比分析**: 按观点分段
```markdown
<!-- SEGMENT: intro -->
<!-- SEGMENT: perspective-a -->
<!-- SEGMENT: perspective-b -->
<!-- SEGMENT: synthesis -->
<!-- SEGMENT: outro -->
```

标记名称用描述性命名，比如 `deep-mobile-agent` 比 `deep-2` 更好。

## 分享心态

视频的核心目的是**让观众从中获得信息和思考**，不是展示你知道多少。

### 信息传递检查

每条精讲必须回答：
1. **这是什么** — 一句话说清楚核心
2. **为什么重要** — 对谁有什么影响
3. **我的判断** — 你怎么看，有什么可行动的

速览必须回答：
1. **这是什么** — 一句话概括
2. **为什么值得知道** — 为什么进了今天的推荐

### 价值密度原则

- 每 30 秒至少输出一个有价值的信息点
- 避免重复说同一个观点的不同表述
- 如果某段没有新信息，砍掉
- 宁可少说一条，也不要每条都蜻蜓点水

## 禁止事项

### 语气

- 恐惧渲染："不做就会被淘汰""时代抛弃你"
- 宏大空泛："颠覆一切""重塑世界""革命性变革"
- 说教："你必须""所有人都该"
- 居高临下："很多人不知道""大部分人忽略了"

### 格式

- 主持人套话："下面请看""让我们来看看"
- 过度副词堆叠
- 机械过渡：每条都用"接下来"
- 千篇一律的开场
- Markdown 格式标记出现在脚本中
- 引号、破折号、括号（脚本和 slides 文字均不出现）
- 中英文/数字之间没加空格
- 复杂长难句未拆分为短句

### 内容

- 脚本总字数低于对应策略的下限
- 逐字复述画面上的文字
- 速览区展开讨论
- 每条精讲都用完全相同的结构和切入角度
- 在脚本中提到文章的评分数字（如"评分 92"），会显得突兀不自然
- 对来源和作者只报名字不介绍背景，观众无法判断可信度

## 脚本自检

写完脚本后检查：

- [ ] **内容准确性**: 数据、结论与源文章一致？没有把推测说成事实？
- [ ] **素材利用**: 源文章中的好图表、好数据、好案例有没有用上？
- [ ] **历史关联**: 与近期内容的关联是否自然？（不强求，有就加）
